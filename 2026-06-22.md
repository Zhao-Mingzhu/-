任务二：基本的命令行管理操作 —— 进程管理与权限隔离（重点）

提示词原文：
在那三个任务中的进程管理（systemctl status/start/stop/restart nginx）...如果我想要测试一下最好是在那个目录来弄不影响后续的操作呢...创建一个新的用户Tester,密码123456来进行进程管理的具体操作，详细步骤。

AI 输出摘要：
1. Nginx 机制解析：解释了 Nginx 作为 Web 服务器和反向代理的核心作用。
2. 安全测试方案：强烈建议不使用 root 直接操作，规划创建普通用户 Tester 并赋予 sudo 权限（加入 wheel 组），以遵循最小权限原则。
3. 进程管理实战：指导通过 sudo systemctl 启停服务，并演示普通用户尝试 kill -9 终结 root 进程时的权限拦截机制。

验证结果：
- 成功创建 Tester 用户并设置密码，切换至该用户进行操作。
- 报错复现1：执行 systemctl restart nginx 报 Failed to restart nginx.service: Unit not found。
- 报错复现2：不加 sudo 执行 systemctl stop nginx，报 Access denied。
- 验证成功：加 sudo 后成功执行服务管理命令；加 sudo kill -9 成功终结进程。
- 注意：kill -9 强制终止后，systemd 可能自动重启 Nginx（与 systemctl stop 不同）。
- 细节观察：发现 systemctl status 在服务未运行时直接退回命令行，服务运行时需按 q 键退出查看界面。

问题与归因：
- 问题1：Unit not found。
- 归因1：服务器纯净系统未预装 Nginx，且 CentOS 7.6 默认软件源不包含 Nginx，需添加 EPEL 源或 Nginx 官方源后安装。
- 问题2：普通用户直接管理系统服务失败。
- 归因2：Linux 权限隔离机制，普通用户无权操作系统级守护进程。
- 问题3：部分命令输出后需按 q，部分直接结束。
- 归因3：Linux 命令分为瞬时执行和接管终端的分页器模式，后者需显式退出。

迭代提示词：
我的里面显示Failed to restart nginx.service: Unit not found...下面应该怎么进行这个任务测试呢？
（AI 响应：指导通过 sudo yum install nginx -y 安装软件，并带领用户完整体验了安装、启动、查看状态(按q退出)、无权限报错、sudo提权成功的全链路。）


任务三：基本的命令行管理操作 —— 环境配置与安全隔离（重点）

提示词原文：
对于任务中的环境配置，我应该也是在/opt里面运行才能够不影响我后续任务的操作吧...上述是不是最好在Tester用户里面弄，还是原始的

AI 输出摘要：
1. 黄金法则确立：明确建议不使用 root 配置环境，确立最小权限原则。root 用于创建系统目录，后续操作移交普通用户。
2. 目录规划与权限交接：指导在 /opt 下创建规范结构（如 /opt/tester_env），并通过 sudo chown -R Tester:Tester /opt/tester_env 将系统目录所有权移交给普通用户。
3. 环境变量配置：指导将环境变量写入用户级配置文件 ~/.bashrc 而非系统级 /etc/profile，确保环境隔离，互不干扰。

验证结果：
- 确立了在 Tester 用户下进行环境配置的绝对优势。
- 掌握了通过 chown 命令变更目录属主的技巧，使得后续在 /opt/tester_env 下的下载、解压操作无需再重复加 sudo。
- 掌握了编辑 ~/.bashrc 并通过 source 命令使环境变量即时生效的标准流程。

问题与归因：
- 问题：如果在 root 下配置环境，存在误删系统文件、环境变量污染、被攻破后风险过高等隐患。
- 归因：未遵循 Linux 系统管理的安全规范，长期使用 root 操作是运维大忌。
- 修正理解：并非完全拒绝 root，而是 root 仅用于创建系统目录，后续操作移交普通用户。

迭代提示词：
上述是不是最好在Tester用户里面弄，还是原始的
（AI 响应：给出了明确的强烈建议在 Tester 用户里弄的结论，并提供了借用权限建目录、移交所有权、用户级配置环境变量的标准操作流。）
